# Agent 开发架构复盘 2

## 1. 本轮检验结论

本轮复盘基于当前工作区代码、关键链路阅读和实际质量命令结果。

检验时间：2026-06-07

质量命令结果：

```text
python -m ruff check .\llm .\tests    通过
python -m pytest -q                   16 passed，存在 .pytest_cache 权限警告
npm.cmd run build                     通过
```

客观结论：

```text
当前项目已经不是简单聊天 Demo，
而是一个具备研发数据查询、LLM Trace、语义指标层雏形的 Agent 原型。

但它仍然不是可上线系统，
更准确的位置是：研发原型后期 / 内部验证前阶段。
```

本轮最大的进步是：

- 后端静态检查通过。
- 后端测试稳定通过 16 个。
- 前端 Vue 工程构建通过。
- Metric Engine 已有单元测试。
- Agent 工具方向开始从碎片工具收敛到 `query_metric`。
- LLM Trace 已经能记录 system prompt、对话、工具调用、工具结果。

本轮最大的风险仍然是：

- Agent 运行时仍是 `create_agent` 自由工具调用，不是 LangGraph 显式流程。
- Admin Trace 接口无鉴权。
- Chat 异常仍统一抛 500。
- 数据库迁移没有真正收敛到 Alembic。
- 语义层只有后端雏形，缺治理页面和完整生命周期。
- 缺 Agent Eval，无法稳定衡量“问法变化时 Agent 是否仍然正确”。

## 3. 本轮关键发现

## 3.1 质量门禁通过，但测试结构还偏窄

当前 16 个测试主要集中在 Metric Engine。

这是必要的，但不充分。

目前缺少：

- `/api/chat` API 测试。
- `/api/chat/stream` SSE 测试。
- `/api/admin/llm-traces` Trace API 测试。
- Dashboard API 测试。
- Trace 写入完整性测试。
- Agent 工具选择测试。
- QueryPlan 生成评测。
- 前端组件测试。

结论：

```text
当前测试证明 Metric Engine 基础可用，
但还不能证明整个 Agent 链路稳定。
```

## 3.2 Agent 仍然依赖 Prompt 约束，而不是流程约束

当前 `llm/agent/llm.py` 使用：

```text
create_agent + tools + system_prompt
```

system prompt 已经明确写了：

```text
指标类查询统一走 query_metric
```

这比碎工具阶段好，但本质仍然是模型自由决策。

风险：

- 模型可能不调用工具。
- 模型可能先回答后查数。
- 模型可能生成错误工具参数。
- 模型可能不知道何时先调用 `list_available_metrics`。
- 多轮复杂查询缺少可控状态。

大厂更稳的做法是：

```text
LangGraph 显式编排
```

建议下一阶段做：

```text
MetricQueryGraph

classify_intent
  -> resolve_time
  -> match_metric
  -> build_query_plan
  -> validate_query_plan
  -> execute_metric
  -> compose_answer
```

优先只改指标查询链路，不要一上来重构所有 Agent 能力。

## 3.3 Metric Engine 是当前最有价值的核心资产

当前 `MetricEngine` 已经做了几件正确的事：

- 校验 metric code。
- 拒绝未知 filter。
- 拒绝未知 group_by。
- 阶段一拒绝跨实体查询。
- 根据语义层编译 SQL。
- LLM 不直接写 SQL。
- 查询后写入 query log。

这符合当前最优方向：

```text
自然语言 -> QueryPlan -> Metric Engine -> 安全 SQL -> 结构化结果
```

但当前仍有明显不足：

- SQL 编译逻辑集中在一个类中，后续会膨胀。
- measure/filter 的表达式处理还偏简单。
- 复杂指标、派生指标、趋势、同比环比暂未形成统一抽象。
- QueryPlan 校验失败后没有自动修复回路。
- compiled SQL 主要用于审计，但还没有在 Trace 页面体系化展示。

建议把 Metric Engine 后续拆成：

```text
MetricValidator
MetricSqlCompiler
MetricExecutor
MetricResultFormatter
MetricQueryLogger
```

但现在不急着拆，等 LangGraph 链路确定后再拆更稳。

## 3.4 Trace 已有价值，但还没有达到治理级别

当前 Trace 已经能记录：

- system prompt。
- human message。
- ai message。
- tool call。
- tool result。
- final output。
- duration。
- status。

这已经能解决用户之前遇到的问题：只存对话、不存 prompt。

但从上线角度，Trace 仍然缺：

- 鉴权。
- 脱敏。
- 保留周期。
- 查询审计。
- 按 trace_id 串联 API 日志。
- QueryPlan 展示。
- compiled SQL 摘要展示。
- tool 参数结构化展示。
- token 统计。
- LLM 错误分类。

结论：

```text
当前 Trace 是调试工具，
还不是生产审计系统。
```

## 3.5 Admin Trace 裸露是 P0 安全问题

当前路由：

```text
/api/admin/llm-traces
/api/admin/llm-traces/{trace_id}
```

没有鉴权。

前端隐藏入口不能算安全。

这是必须明确指出的问题：

```text
隐藏路由不是权限控制。
只要接口能被访问，就等于裸露。
```

最低可接受方案：

```text
后端增加 X-Admin-Token 校验
前端请求 admin 接口时带 token
admin 路由进入前做 guard
token 放 .env，不写死到代码
```

长期方案：

```text
用户登录
JWT / Session
RBAC
数据权限
操作审计
```

## 3.6 Chat API 的错误模型还不成熟

当前 `/api/chat` 捕获异常后：

```text
HTTPException(status_code=500, detail=str(e))
```

问题：

- 用户侧只能看到 500。
- 前端难以区分 LLM 超时、DB 错误、工具参数错误、指标不存在。
- 可能泄露内部异常信息。
- 不利于 Trace 和前端提示联动。

建议建立统一错误模型：

```text
{
  "error_code": "INVALID_QUERY_PLAN",
  "message": "当前问题无法匹配到可用指标",
  "trace_id": "...",
  "retryable": false
}
```

优先错误码：

```text
LLM_CALL_FAILED
TOOL_CALL_FAILED
INVALID_QUERY_PLAN
METRIC_NOT_FOUND
DB_QUERY_FAILED
PERMISSION_DENIED
TRACE_NOT_FOUND
```

## 3.7 本地项目卫生需要处理

本轮检查发现：

```text
rg --files
git status
pytest
```

都提示 `.pytest_cache` 或 `pytest-cache-files-*` 目录存在权限问题。

这不影响当前测试通过，但会影响：

- 搜索体验。
- git 状态可读性。
- CI 一致性。
- 后续工具扫描。

建议：

- 确认这些目录是否可删除。
- 把 pytest 临时目录加入 `.gitignore`。
- 修复 `.pytest_cache` 权限或删除后重建。

注意：不要用暴力删除命令处理，先确认目录归属和权限。

## 4. 当前项目距离上线还差什么

按优先级分：

## 4.1 P0：不上这些不能上线

- 后端鉴权。
- Admin Trace 权限保护。
- 数据库账号最小权限，不使用 root 作为应用账号。
- 统一异常处理。
- Trace 脱敏。
- Alembic 正式迁移。
- 基础 API 集成测试。
- 生产配置与本地配置隔离。
- `.env` 保持不入库。

## 4.2 P1：不上这些不能稳定内部试用

- LangGraph MetricQueryGraph。
- QueryPlan 校验和修复回路。
- Agent Eval 数据集。
- Trace 展示 QueryPlan 和工具参数。
- Metric Catalog 只读页面。
- Dashboard API 测试。
- SSE 异常和断线处理。
- 前端错误态、加载态、空态。

## 4.3 P2：上这些才像平台

- 指标版本管理。
- 指标审批流。
- 指标血缘。
- 指标使用统计。
- 多租户或数据域隔离。
- Prompt 版本管理。
- 模型路由。
- 缓存层。
- 任务队列。
- CI/CD。
- Docker Compose。

## 5. 下一轮建议路线

## 5.1 第一优先级：MetricQueryGraph

不要继续加碎工具。

下一步最值得做的是：

```text
用 LangGraph 重构指标查询链路。
```

建议目标：

```text
用户问：最近一个月创建了多少 Bug？其中多少已关闭？关闭率多少？

Graph 输出：
1. 识别为 metric_query
2. 解析时间范围
3. 匹配指标 bug_count / closed_bug_count / bug_close_rate
4. 生成 QueryPlan
5. 校验 QueryPlan
6. 执行 MetricEngine
7. 输出带口径的回答
8. Trace 保存完整过程
```

## 5.2 第二优先级：Admin 鉴权

先做最小版本即可：

```text
X-Admin-Token
```

保护：

```text
/api/admin/*
```

后续再升级完整登录系统。

## 5.3 第三优先级：Trace 增强

Trace 下一步要补：

```text
query_plan
validated_query_plan
validation_error
compiled_sql
metric_result
tool_args
error_code
```

前端 Trace 页面要能看：

```text
模型收到了什么
模型决定调用什么
工具参数是什么
工具返回什么
最终回答如何生成
```

## 5.4 第四优先级：Agent Eval

建立 `tests/evals/metric_questions.json`。

每条记录包括：

```text
question
expected_intent
expected_metric_codes
expected_time_range
expected_filters
expected_group_by
expected_tool
```

先写 20 条，不追求多，先覆盖高频问法。

## 5.5 第五优先级：Metric Catalog 页面

先做只读页面：

```text
/admin/metrics
/admin/metrics/:metric_code
```

展示：

- 指标名称。
- 指标编码。
- 统计口径。
- 公式。
- 可用维度。
- 可用过滤条件。
- 负责人。
- 状态。

不要马上做复杂编辑器。

## 6. 本轮评分

按大厂内部应用落地标准估算：

```text
原型完整度：72%
工程化程度：58%
Agent 可控性：45%
语义层成熟度：52%
```

评分说明：

- 工程化程度上升，因为 Ruff、pytest、build 通过。

## 7. 客观结论

当前项目方向是正确的：

```text
研发数据 Agent
  + 语义指标层
  + 可审计 Trace
  + 前后端工程化
```

但当前还没有达到上线级别。

下一步最优解不是继续堆功能，而是补四个关键支柱：

```text
1. LangGraph 显式流程
2. Admin 鉴权
3. Agent Eval
4. Trace + QueryPlan 全链路审计
```

如果下一轮完成这四件事，项目会从：

```text
研发原型后期
```

进入：

```text
内部试用准备阶段
```

这是当前最符合架构演进、维护性和大厂实践的路线。

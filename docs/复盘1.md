# Agent 开发架构复盘 1

## 1. 结论

当前项目已经从“简单 LLM 对话 Demo”演进到了“研发数据 Agent 原型”阶段，具备了前端工作台、后端 API、工具调用、MySQL 业务数据、LLM Trace、组织层级和 Metric Layer 雏形。

但如果站在大厂实际落地 Agent 应用的标准看，当前系统仍处于：

```text
技术验证阶段 / 原型阶段
```

还没有达到：

```text
可灰度试用
可内部上线
可生产上线
```

主要原因不是“功能少”，而是 Agent 系统最关键的几层能力还不完整：

- Agent 编排层仍偏单体 `create_agent`。
- 工具治理不足，工具边界和工具元数据不够系统。
- Metric Layer 已有方向，但离稳定语义层还有距离。
- 缺少权限、安全、审计、评测、降级、监控和部署闭环。
- 测试覆盖不足，无法保证工具和指标查询稳定。

当前项目价值很高，但它现在更像一台正在改装中的发动机，还不能直接上路载客。

## 3. 当前主要不足

## 3.1 Agent 编排层不足

### 现状

当前 Agent 主要依赖：

```text
LangChain create_agent
```

工具直接挂在 Agent 上，由模型自由选择。

### 问题

这种方式适合原型，但大厂真实业务 Agent 通常不会长期只靠一个通用 Agent 完成全部流程。

当前缺少：

- 明确的意图识别节点。
- 时间解析节点。
- QueryPlan 生成节点。
- QueryPlan 校验节点。
- 工具调用节点。
- 结果解释节点。
- 失败恢复节点。
- 人工确认节点。

如果所有逻辑都让 `create_agent` 自由处理，会出现：

- 调错工具。
- 参数不稳定。
- 口径不一致。
- 错误难复盘。
- 无法做流程级测试。

### 大厂常见做法

企业级 Agent 更常见的是：

```text
Router / Planner / Executor / Critic / Guardrail
```

或者用 LangGraph 建明确状态图：

```text
receive_input
  -> classify_intent
  -> resolve_time_range
  -> build_query_plan
  -> validate_query_plan
  -> execute_tool
  -> compose_answer
  -> trace_result
```

### 优化方案

下一阶段应从单体 Agent 升级为 LangGraph 工作流：

```text
Chat Agent
Metric Query Agent
Metric Definition Agent
Admin Trace Agent
```

第一步只做 Metric Query Graph：

```text
用户问题
  -> 指标识别
  -> 时间解析
  -> 过滤条件解析
  -> QueryPlan 生成
  -> QueryPlan 校验
  -> query_metric
  -> 回答生成
```

## 3.2 工具体系不足

### 现状

项目里已经出现多个工具：

- Bug 指标工具。
- 用户指标工具。
- 组织查询工具。
- 时间工具。
- Metric 查询工具。
- Metric 元数据工具。

### 问题

工具正在变多，但还没有形成工具治理规范。

当前缺少：

- 工具分级。
- 工具权限。
- 工具输入输出标准。
- 工具超时。
- 工具错误码。
- 工具调用审计。
- 工具是否只读/可写的标识。
- 工具评测样例。

继续发展下去会变成：

```text
工具越写越多，模型越来越不知道该选哪个。
```

### 大厂常见做法

大厂不会把所有能力都平铺成工具，而会做工具分层：

```text
基础工具
业务工具
语义工具
执行工具
管理工具
```

例如：

```text
query_metric              高阶语义工具
get_metric_definitions    元数据工具
resolve_org               组织解析工具
update_bug_status         写操作工具，需确认
```

### 优化方案

建立工具注册表：

```text
tool_code
tool_name
tool_type
read_only
requires_confirmation
allowed_roles
input_schema
output_schema
timeout_ms
description
examples
```

并减少碎工具，收敛到高阶工具：

```text
query_metric
list_metric_definitions
resolve_entity
resolve_dimension_value
```

## 3.3 Metric Layer 仍是雏形

### 现状

项目已经有语义层文档，并出现了：

- `metric_layer_repository`
- `metric_engine`
- `metric_query_tools`
- `metric_meta_tools`
- `metric_query` schema

这说明方向是对的。

### 问题

当前 Metric Layer 仍存在几个风险：

- Entity / Dimension / Measure / Metric 的边界需要进一步稳定。
- QueryPlan 校验需要系统化。
- Join 规则需要白名单化。
- 指标公式需要版本管理。
- 指标口径还没有前端管理页面。
- 指标创建 Agent 还没有形成审核流程。
- 指标查询结果还没有统一 Result Schema。

### 大厂常见做法

语义层一般会拆成：

```text
Entity Layer
Dimension Layer
Measure Layer
Metric Layer
Access Policy Layer
Query Engine
```

并且所有查询都走：

```text
Natural Language
  -> QueryPlan
  -> Validate
  -> Compile SQL
  -> Execute
  -> Explain
```

### 优化方案

明确 Metric Layer 的核心契约：

```text
LLM 只能输出 QueryPlan。
Metric Engine 只接受 QueryPlan。
SQL 只由 Metric Engine 编译。
所有字段必须来自语义层定义。
所有 join 必须来自 join_config 白名单。
```

第一阶段只支持：

```text
bug entity
count measure
ratio metric
time_range
filters
group_by
```

不要过早支持复杂表达式。

## 3.4 记忆体系不足

### 现状

Agent 使用 session 内存保存最近消息：

```text
self.sessions[session_id] = messages[-20:]
```

LLM Trace 会保存调用快照。

### 问题

当前把“上下文记忆”和“Trace 追踪”混在同一发展阶段，但二者职责不同：

```text
Chat Memory：给 Agent 使用。
LLM Trace：给开发排查使用。
Business Memory：保存用户偏好和业务口径。
```

当前缺少：

- 会话表。
- 会话消息表。
- 用户偏好表。
- 业务口径记忆。
- 记忆压缩策略。
- 记忆失效策略。

### 大厂常见做法

通常会分为：

```text
Short-term Memory
Long-term Memory
User Preference Memory
Business Rule Memory
Trace Log
```

### 优化方案

新增：

```text
chat_sessions
chat_messages
agent_memories
business_glossary
```

并明确：

```text
Trace 允许重复保存完整上下文快照。
Memory 不应重复保存同一消息。
业务口径应进入 glossary 或 metric definitions。
```

## 3.5 安全与权限不足

### 现状

当前接口包括：

```text
/api/chat
/api/admin/llm-traces
```

但没有完整认证鉴权。

### 问题

上线风险很高：

- Admin Trace 接口可查看 system prompt 和用户输入。
- LLM Key 存在本地 `.env`。
- 数据库使用 root 用户。
- CORS 仍偏开发配置。
- 没有用户身份。
- 没有组织数据权限。
- 没有接口限流。

### 大厂常见做法

至少需要：

```text
Authentication
RBAC
Data Scope
Audit Log
Secret Management
Rate Limit
PII Masking
```

### 优化方案

P0 必须补：

- 登录认证。
- 用户角色。
- Admin 接口鉴权。
- 数据库应用账号，禁止 root。
- Trace 脱敏。
- CORS 生产配置。
- API 限流。

角色建议：

```text
admin
developer
analyst
viewer
```

## 3.6 可观测性不足

### 现状

已有 LLM Trace。

### 问题

LLM Trace 只是可观测性的一部分。当前缺少：

- 请求级 trace_id。
- API latency。
- Tool latency。
- SQL latency。
- LLM token usage。
- LLM cost。
- 错误分类。
- 指标查询日志。
- 慢查询记录。
- 告警。

### 大厂常见做法

通常需要三类观测：

```text
Logs
Metrics
Traces
```

Agent 还需要：

```text
Prompt Trace
Tool Trace
QueryPlan Trace
LLM Cost Trace
Evaluation Trace
```

### 优化方案

新增统一 Trace 上下文：

```text
trace_id
session_id
request_id
user_id
tool_call_id
metric_query_id
```

并记录：

```text
每次模型调用
每次工具调用
每次 SQL 执行
每次 QueryPlan 编译
每次错误
```

## 3.7 测试与评测不足

### 现状

当前测试很少，主要是基础 app 测试。

### 问题

Agent 项目上线前不能只测 API 是否启动。

缺少：

- Tool 单测。
- Repository 单测。
- Metric Engine 单测。
- QueryPlan 校验测试。
- Agent 意图识别测试。
- Prompt 回归测试。
- 前端组件测试。
- 端到端测试。
- 数据集评测。

### 大厂常见做法

Agent 评测至少包括：

```text
Golden Dataset
Tool Call Accuracy
Answer Correctness
SQL Safety
Latency
Cost
Regression
```

### 优化方案

建立 `evals/`：

```text
evals/metric_queries.yaml
evals/tool_calls.yaml
evals/expected_answers.yaml
```

每次改 Prompt 或工具后跑评测：

```text
问题 -> 期望 QueryPlan -> 期望工具 -> 期望结果
```

## 4. 和大厂 Agent 应用的差距

如果用大厂落地标准衡量，当前差距主要在以下层面：

| 层面 | 当前状态 | 大厂常见要求 |
|---|---|---|
| Agent 编排 | 单体 Agent 为主 | LangGraph/Workflow 显式编排 |
| 工具体系 | 工具逐步增加 | 工具注册、权限、审计、评测 |
| 指标查询 | Metric Layer 雏形 | 语义层 + QueryPlan + Query Engine |
| 数据权限 | 基本没有 | 用户、角色、组织数据权限 |
| Trace | 有 LLM Trace | Prompt/Tool/SQL/Cost 全链路 Trace |
| 记忆 | session 内存 | 分层 Memory + 业务口径沉淀 |
| 测试 | 很少 | 单测、集成、E2E、Agent Eval |
| 部署 | 本地启动 | Docker/CI/CD/迁移/监控 |
| 安全 | 开发态 | 鉴权、脱敏、限流、密钥管理 |

## 6. 优先级路线

### P0：先让项目稳定

- 修复 Ruff 全部问题。
- 增加关键测试。
- 固化数据库迁移。
- 给 Admin Trace 加权限。
- 数据库改应用账号。
- 完善统一错误响应。

### P1：完成 Metric Layer 闭环

- 完善 Entity/Dimension/Measure/Metric 表。
- 完成 `query_metric` 工具。
- 完成 Metric Engine 校验。
- 让 Agent 查询指标只走 QueryPlan。
- Trace 记录 QueryPlan 和工具结果。

### P2：升级 Agent Runtime

- 引入 LangGraph 显式工作流。
- 拆出 Metric Query Graph。
- 拆出 Metric Definition Graph。
- 加人工确认节点。
- 加失败恢复节点。

### P3：产品化与上线准备

- 登录认证。
- 权限路由。
- 指标管理页面。
- Docker 化。
- CI/CD。
- 日志、指标、告警。
- Agent Eval 数据集。

## 7. 下一步最建议做什么

不要继续盲目加工具。下一步应该做三件事：

```text
1. 让现有代码质量检查全部通过。
2. 把 query_metric 做成唯一指标查询入口。
3. 用 LangGraph 拆出 Metric Query 工作流。
```

具体顺序：

```text
第一步：修 Ruff 问题。
第二步：补 Metric Engine 单测。
第三步：补 QueryPlan Schema。
第四步：让 Agent 输出 QueryPlan 而不是自由调多个工具。
第五步：Trace 里记录 QueryPlan、工具调用、SQL 摘要。
```

## 8. 最终评价

当前项目方向正确，尤其是从碎工具转向 Metric Layer，这是很重要的一步。

但从架构成熟度看：

```text
Agent 原型能力：中等
工程化能力：初级到中级
大厂上线能力：不足
可扩展方向：正确
当前稳定性：不足
```

如果继续沿着：

```text
LangGraph 工作流
Metric Layer
Tool Governance
LLM Trace
Agent Eval
权限与安全
```

这条路线走，项目会逐步接近真实企业 Agent 应用。

如果继续随需求增加碎工具，则项目很快会难以维护。

当前最关键的架构选择是：

```text
让 Agent 少“自由发挥”，多“生成结构化计划”。
让后端少“临时 SQL”，多“语义层编译”。
让工具少“碎片能力”，多“稳定业务接口”。
```
